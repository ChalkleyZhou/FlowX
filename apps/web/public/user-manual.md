# FlowX 用户使用手册

本文档面向产品、研发负责人、项目经理和业务协作同学，帮助你在 FlowX 中从需求录入一路推进到 AI 执行、人工评审、问题沉淀与发布。

## 1. 你会在 FlowX 里做什么

FlowX 把研发流程拆成可中断、可确认的阶段，核心目标是：

- 让 AI 先产出，再由人确认关键节点
- 每一阶段都可追溯，降低黑盒风险
- 统一管理需求、代码仓库、执行结果、审查结论

主流程是：

`工作区 -> 项目 -> 需求 -> 需求定稿 -> 工作流执行 -> AI 审查 -> 人工决策 -> 问题/缺陷沉淀（可选部署）`

## 2. 功能导航说明

左侧导航主要有以下模块：

- `工作区`：管理业务边界与代码仓库
- `项目`：承接实际交付单元
- `需求`：录入需求、验收标准，发起流程
- `工作流`：查看和推进每条研发流程
- `问题项`：沉淀 AI 审查里的改进项（Issue）
- `缺陷`：沉淀明确 Bug（Bug）
- `AI 凭据`：配置你个人的 Cursor / Codex 凭据

## 3. 第一次使用（10 分钟上手）

### 步骤 1：登录系统

在登录页可选两种方式：

- 账号密码登录（也支持注册后登录）
- 钉钉登录（适合企业组织身份接入）

### 步骤 2：配置个人 AI 凭据（推荐）

进入 `AI 凭据` 页面，按需配置：

- `Cursor API Key`
- `Codex / OpenAI API Key`

保存后，系统在工作流执行时会优先使用你的用户凭据。

### 步骤 3：创建工作区并收录仓库

进入 `工作区` 页面：

1. 新建工作区（例如“支付平台”）
2. 在工作区下新增代码仓库（填写仓库名称与地址）
3. （可选）配置仓库默认分支与部署模板

> 建议先把真实会改动的仓库纳入工作区，后续需求和工作流才能准确拿到上下文。

### 步骤 4：创建项目

进入 `项目` 页面，新建项目并选择归属工作区。

一个工作区下可以有多个项目；一个项目下可以有多条需求。

### 步骤 5：创建需求

进入 `需求` 页面，点击“新增需求”，填写：

- 所属项目
- 需求标题
- 需求描述
- 验收标准
- 影响仓库范围（可留空，留空表示继承工作区默认仓库）

## 4. 从需求到工作流：标准操作流程

### 4.1 在需求详情先完成“定稿”

打开某条需求详情后，会看到 ideation 三步：

1. 头脑风暴
2. UI 设计
3. 定稿

只有当需求状态达到“已定稿”后，建议再启动研发工作流。定稿会把确认后的产品简报合并回需求描述，避免执行阶段上下文不一致。

### 4.2 启动工作流

可以在需求列表或需求详情里启动工作流。启动时可选本次执行的仓库范围：

- 不选：使用需求默认仓库范围
- 手动选择：只推进本次要并行处理的仓库

### 4.3 在工作流详情推进每个阶段

工作流核心阶段通常是：

1. 任务拆解（Task Split）
2. 技术方案（Technical Plan）
3. 开发执行（Execution）
4. AI 审查（AI Review）
5. 人工评审（Human Review）

在各阶段你可以进行的常见操作：

- 执行阶段（如“执行任务拆解”“执行开发”“执行 AI 审查”）
- 确认 / 驳回（用于需要人工把关的节点）
- 人工修改（直接修正阶段产出）
- 填写意见（用于退回或指导下一轮）

AI 审查完成后，人工决策通常有三种：

- `通过`：工作流进入完成态
- `退回开发执行`：回到执行阶段继续修复
- `回滚`：终止当前结果并回退

### 4.4（可选）提交并推送、触发部署

当人工确认通过后，可在工作流内执行：

1. `提交并推送到远程`
2. `按仓库触发部署`

部署会读取仓库级部署模板。如果按钮不可用，通常是仓库未开启部署配置或当前工作流无可部署仓库。

## 5. 状态怎么看

### 工作流状态（常见）

- `待任务拆解` / `待确认任务拆解`
- `待生成方案` / `待确认技术方案`
- `待执行开发` / `开发执行中`
- `待 AI 审查`
- `待人工评审`
- `已完成` / `失败`

### 阶段状态（常见）

- `待执行`
- `执行中`
- `已完成`
- `待人工确认`
- `执行失败`
- `已驳回`

## 6. 问题沉淀与追踪

- 在 `问题项` 页面追踪改进项（Issue）
- 在 `缺陷` 页面追踪明确故障（Bug）
- 每条记录都可追溯到来源需求、工作流和审查上下文，便于复盘和闭环

## 7. 常见问题

### Q1：为什么我不能直接“启动研发工作流”？

通常是需求还未完成“定稿”。请先进入需求详情完成头脑风暴/设计确认并定稿。

### Q2：为什么“触发部署”失败或不可用？

优先检查：

- 当前工作流是否有可部署仓库
- 对应仓库是否在工作区里配置了部署模板并启用
- 是否已先执行“提交并推送到远程”

### Q3：我配置了 AI 凭据，为什么没生效？

请确认：

- 你当前登录账号与配置凭据的是同一账号
- 页面状态显示“已配置”
- 工作流是凭据配置后新启动的（旧流程可能仍使用旧上下文）

## 8. 推荐团队使用顺序

建议团队按下面顺序落地：

1. 管理员先完成工作区、仓库、部署模板初始化
2. 各成员配置自己的 `AI 凭据`
3. 项目负责人按项目组织需求
4. 需求先定稿，再发起工作流
5. 在 AI 审查后坚持人工决策与问题沉淀

这样可以把“AI 产出”和“人类决策”稳定地串成可复用流程。
