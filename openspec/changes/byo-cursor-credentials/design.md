## Context

FlowX 当前在服务端容器内执行 Cursor CLI。认证来源是进程级环境变量 `CURSOR_API_KEY` 或者容器中的登录态，缺少用户级隔离。系统已有会话身份（`UserSession`）和工作流发起链路，可用于将发起用户上下文传递到 AI 执行层。该变更需要在不破坏现有部署的前提下引入按用户凭据执行能力，并且满足日志脱敏、最小暴露面与可审计要求。

## Goals / Non-Goals

**Goals:**
- 支持每个用户独立配置 Cursor API Key，并在执行时按发起用户注入。
- 服务端仅保存加密密文；明文仅用于瞬时解密并传入子进程环境。
- 兼容现有实例级 `CURSOR_API_KEY` 与 CLI 登录态回退，避免一次性破坏生产环境。
- 提供清晰错误语义：未配置凭据、凭据无效、解密失败、CLI 认证失败。

**Non-Goals:**
- 不改造 Codex provider 的认证模型。
- 不引入外部 KMS/Vault（本次先使用应用内主密钥加密）。
- 不变更工作流阶段编排与状态机语义。

## Decisions

1. **新增用户级凭据表（按 userId + provider 唯一）**
   - 方案：在 Prisma 新增 `UserAiCredential`（`userId`、`provider`、`encryptedSecret`、`keyVersion`、时间戳）。
   - 原因：满足多用户隔离与扩展到其他 provider 的可能性。
   - 备选：把密文塞进 `User` 扩展字段。未选原因是扩展性和审计性差。

2. **应用内对称加密，主密钥来自环境变量**
   - 方案：新增 `CredentialCryptoService`，使用 `FLOWX_CREDENTIAL_MASTER_KEY` 执行加解密；密文使用可版本化格式。
   - 原因：快速落地、部署简单、便于后续密钥轮换。
   - 备选：直接明文存库。未选原因是违反安全要求。

3. **执行时注入 per-invocation env，而非全局 process.env 变更**
   - 方案：`CursorAiExecutor` 允许接收请求级环境覆盖，并在 `spawn` 时合并。
   - 原因：避免并发污染、避免跨请求凭据串用。
   - 备选：修改全局 `process.env.CURSOR_API_KEY`。未选原因是线程/并发不安全。

4. **凭据优先级固定为 用户 > 实例级 > 登录态**
   - 方案：运行时先查用户密钥；缺失时回退 `process.env.CURSOR_API_KEY`；再回退 CLI 登录态。
   - 原因：兼容旧部署并支持渐进迁移。
   - 备选：强制必须用户密钥。未选原因是上线风险高。

5. **接口返回凭据状态，不返回明文**
   - 方案：提供 `configured: boolean`、`updatedAt` 等状态字段；删除后清空。
   - 原因：满足前端可用性，同时杜绝明文泄漏。

## Risks / Trade-offs

- **主密钥泄漏风险** → 使用独立环境变量、最小权限注入、运维轮换流程与审计日志。
- **历史部署未配置主密钥** → 在启动时明确检查并给出可操作报错；支持只读兼容模式。
- **并发执行下环境污染风险** → 仅在 `spawn` 入参覆盖环境，禁止写全局环境变量。
- **用户凭据失效导致失败率上升** → 细化错误码并提供前端引导，让用户快速重新配置。
- **回退链路掩盖配置问题** → 在日志中记录使用了哪一层凭据来源（不含敏感值）。

## Migration Plan

1. 数据库迁移：新增 `UserAiCredential` 表与必要索引。
2. 发布后端 API：先支持凭据管理，不改变默认执行行为。
3. 接入执行链路：启用“用户优先、实例回退”逻辑。
4. 前端发布配置入口：引导用户自助配置。
5. 文档更新与灰度策略：先兼容模式，后续可加开关禁止实例级回退。

回滚策略：
- 代码回滚后仍保留新表，不影响旧逻辑。
- 出现问题可临时仅使用实例级 `CURSOR_API_KEY`，关闭用户凭据读取路径。
