# FlowX 产品与交互体验评审

> 基于 2026-07-23 仓库源码、前端规范和后端接口的产品审查。本文是产品与交互基线，不是某个页面的视觉改稿说明。

## 结论先行

FlowX 不应该继续作为一组“需求、工作流、简报、缺陷、排期”的页面集合演进，而应该明确为：

> 面向研发团队的 AI 流程控制平面：把待办、上下文、阶段决策、执行证据和质量结果串成可追踪的交付链路。

当前最影响体验的不是颜色或卡片样式，而是三个产品级问题：

1. 用户进入系统后不知道“现在最该处理什么”。
2. 列表没有统一查询模型，数据规模变大后会慢、不可分享、不可恢复。
3. 工作流详情同时承担导航、执行、审查和发布，主任务不突出，状态和操作分散。

建议先建立统一的信息架构和交互基座，再逐页改造，而不是继续在每个页面补局部状态。

## 1. 产品模型

### 1.1 用户真正需要的不是页面

FlowX 的核心对象关系应在界面上保持稳定：

```text
工作区
  -> 项目
    -> 需求
      -> 工作流运行
        -> 阶段
          -> Artifact / Evidence / Decision
            -> Issue / Bug / 发布
```

页面只是这些对象的不同工作视图：

- 需求：待进入研发的输入和优先级队列。
- 工作流：正在推进的流程和人工门禁。
- 项目：上下文、仓库、需求和交付健康度。
- 质量：AI Review、Issue、Bug 的处置队列。
- 排期：人员和需求的时间承诺。
- 简报和每日 Code Review：周期性信息消费，不应抢占研发主路径。

### 1.2 首屏应该是“我的工作队列”

登录后的默认入口建议从“需求列表”升级为工作队列，直接展示：

- 待我确认：设计、Demo、任务拆解、技术方案、人工审查。
- 执行中：我发起或我负责的本地/云端执行。
- 失败和阻塞：需要重试、补配置或人工介入的流程。
- 最近更新：我关注的需求、工作流、审查结果。
- 项目健康概览：仅保留少量总览指标，不做装饰性 dashboard。

这能把“查看数据”转成“完成工作”。工作流、审查和排期页面都应能从队列直接跳入，并保留返回位置。

### 1.3 导航建议

当前侧边栏把 9 个业务入口平铺，设置入口又隐藏在用户菜单中。建议按用户任务分组：

```text
研发
  工作队列
  需求
  工作流
  项目

质量与交付
  审查
  问题与缺陷
  排期

信息
  项目简报
  每日 Code Review

管理
  工作区与代码库
  组织成员
  AI / Git 凭据
  投递目标
```

“问题项”和“缺陷”在导航上建议合并为“问题与缺陷”，在页面内用 Tabs 区分；工作流 AI Review 和每日 Code Review 要保持明确边界，但可以在质量中心统一入口展示。

## 2. 列表页统一体验

### 2.1 现状证据

- `apps/web/src/pages/WorkflowRunsPage.tsx` 一次加载工作区、项目、需求和全部工作流，在浏览器内过滤；API `GET /workflow-runs` 当前只有 `runType` 参数。
- `apps/web/src/pages/RequirementsPage.tsx`、`ProjectsPage.tsx`、`OrganizationUsersPage.tsx` 仍然完整加载后本地筛选，未提供统一分页。
- `IssuesPage.tsx` 和 `BugsPage.tsx` 有分页，但只是对全量结果 `slice`，固定每页 8 条；页码、筛选和关键词没有进入 URL，也没有持久化。
- `BriefingsPage.tsx` 有独立的 `localStorage` 偏好；`RequirementsPage.tsx` 只持久化 AI provider。筛选保存不是通用能力。
- `apps/web/src/api.ts` 的列表 helper 返回数组，尚无统一的分页元数据契约。
- 后端 `findMany` 列表接口普遍没有 `skip/take`、搜索、排序和 `count`。

### 2.2 统一列表契约

第一阶段采用 offset pagination，避免一次引入 cursor 的复杂度。所有可增长列表统一返回：

```ts
interface PageResult<T> {
  items: T[];
  page: number;
  pageSize: number;
  total: number;
  totalPages: number;
}

interface ListQuery {
  page?: number;
  pageSize?: 20 | 50 | 100;
  q?: string;
  sort?: string;
  order?: 'asc' | 'desc';
}
```

业务筛选作为显式字段加入 query DTO，例如 `workspaceId`、`projectId`、`status`、`runType`。后端负责过滤、排序、分页和总数，前端不再拉全量数据后 `filter/slice`。

以下规则应成为所有列表的默认行为：

- 默认 `pageSize=20`，提供 20 / 50 / 100。
- 默认排序按 `updatedAt desc`；并列时用 `id` 保证稳定顺序。
- 搜索输入 300ms debounce；搜索、筛选、排序变化自动回到第 1 页。
- URL 保存当前查询状态，支持刷新、复制链接、浏览器前进后退和详情返回。
- `localStorage` 只保存“个人默认偏好”，不覆盖 URL；按 `organizationId + userId + pageKey` 隔离。
- 空结果分为“还没有数据”和“筛选无结果”，后者提供“清除筛选”。
- loading 使用保持布局的 skeleton；翻页时不要让整个页面跳回顶部或闪烁空白。
- 列表底部同时显示“第 x-y 条，共 z 条”、页码、每页数量和上一页/下一页。
- 移动或窄屏优先保证查询和主操作可用，次级列收进详情，不把完整桌面表格硬挤成多行卡片。

### 2.3 筛选持久化的三层边界

不要再让每个页面自己决定“存不存”：

| 状态 | 来源 | 生命周期 | 是否可分享 |
| --- | --- | --- | --- |
| 当前搜索、筛选、排序、分页 | URL query | 当前视图 | 是 |
| 上次使用的 workspace、project、pageSize、视图模式 | `localStorage` | 用户级 | 否 |
| 表单草稿、弹窗开关、加载态 | React state | 当前会话 | 否 |

敏感凭据、密码、token、AI/Git secret 和一次性操作绝不进入筛选偏好。

### 2.4 列表页改造优先级

| 页面 | 当前主要问题 | 建议的主查询 | 优先级 |
| --- | --- | --- | --- |
| 工作流 | 全量加载；筛选维度多；无法定位待处理 | 状态、类型、工作区、项目、需求、负责人、更新时间 | P0 |
| 需求 | 全量加载；工作流状态混在详情数据中 | 关键词、项目、状态、优先级、更新时间 | P0 |
| 问题与缺陷 | 仅前端分页；分页状态易丢失 | 关键词、工作区、状态、优先级/严重级别、来源 | P0 |
| 项目 | 全量加载；缺少搜索和排序 | 关键词、工作区、活跃状态、健康度 | P1 |
| 工作区/代码库 | 一个工作区内嵌所有仓库，页面容易过长 | 工作区列表；代码库作为可展开详情或独立子列表 | P1 |
| 组织成员 | 有搜索但没有分页 | 关键词、角色、账号状态 | P1 |
| 简报 / 每日 Code Review | 当前按项目加载，历史量会增长 | 项目、周期/日期、状态 | P1 |
| 投递目标 / 数据源 | 管理型列表，需保留项目上下文 | 项目、类型、启用状态 | P2 |

## 3. 工作流详情重构

### 3.1 现状判断

`WorkflowRunDetailPage.tsx` 超过 3000 行，集中管理阶段选择、状态轮询、本地 Agent、OpenDesign、Demo 预览、计划 Artifact、执行证据、Diff、审查 findings、回滚、删除、发布和部署。页面存在 8 个阶段卡片、多个状态分支和大量操作按钮，导致：

- 用户需要滚动很长距离才能找到当前阶段的内容和下一步。
- “运行”“确认”“驳回”“反馈”“本地启动”“完成本地执行”等动作分散在不同区域。
- 当前状态、当前阶段、阶段历史、执行会话和最终结果没有明确层次。
- `WorkflowSteps` 将阶段做成 4 列网格，更像目录卡片，不像流程控制器。
- 详情页的阶段选择和反馈状态主要存在 React state 中，刷新后无法恢复用户正在看的上下文。

这不是继续微调间距能解决的问题，需要从“长页面”改成“当前任务工作台”。

### 3.2 推荐页面结构

```text
Breadcrumb: 工作区 / 项目 / 需求 / 工作流
DetailHeader: 需求标题 + 工作流类型 + 总状态 + 最近更新时间
ActionBar: 当前唯一主操作 + 次级动作 + 更多菜单
StageRail: 8 个阶段的紧凑时间线，显示完成/进行中/待确认/失败

Main workspace
  当前阶段（默认）
    阶段目标、输入上下文、产物、执行状态、下一步
  变更与审查
    Diff、文件、Review findings、Issue/Bug 处置
  上下文
    需求、仓库、分支、任务、技术方案
  历史
    状态变化、执行会话、人工决策、Artifact/Evidence

Sticky side rail
  当前状态
  阶段完成度
  阻塞原因
  负责人 / 执行器 / 更新时间
```

阶段栏默认只展示阶段名和状态，点击后切换 `stage` query；内容区一次只展示一个阶段。`tab` 和 `stage` 都进入 URL，使刷新、分享和返回可恢复。

### 3.3 “一个状态，一个主动作”

每个状态只允许有一个最主要的下一步，其他动作降级为次级或“更多”：

| 状态类型 | 页面应该回答的问题 | 主动作示例 |
| --- | --- | --- |
| Pending | 谁来启动、启动什么 | 开始生成 / 领取本地执行 |
| Running | 当前是否正常、何时完成 | 查看执行详情；运行中不重复提供启动 |
| Waiting confirmation | 我需要确认什么 | 确认并继续 |
| Failed | 为什么失败、如何恢复 | 查看失败原因 / 重试 |
| Human review | 哪些发现必须处理 | 处理 findings / 通过审查 |
| Done | 交付是否完成 | 发布变更 / 创建部署 |

按钮文案要表达结果而不是内部状态。比如“确认任务拆解并生成技术方案”比“Confirm”更可执行；禁用按钮旁边必须解释“等待本地 Agent 回传完成报告”等原因。

### 3.4 阶段内容的产品化

- 需求上下文、AI 头脑风暴、设计和 Demo：以“产物预览 + 人工决策”呈现，反馈输入紧贴决策按钮，不放到页面末尾。
- 任务拆解和技术方案：以可扫描的任务表/方案摘要为主，文件、风险和完整 HTML 作为可展开内容。
- 云端执行：展示执行进度、当前阶段、日志摘要、变更统计和刷新时间；长日志默认折叠。
- 本地执行：展示一张明确的“本地执行任务卡”，包含领取、工作分支、仓库、完成条件和回传状态；不要把本地 Agent 配置细节和业务结果混在一起。
- Diff Review：左侧文件列表、右侧 diff，顶部固定仓库/分支切换；审查结论和 findings 在同一上下文中完成。
- Review findings：按阻断级别、严重程度、来源分组；每条 finding 提供接受、忽略、转 Issue、转 Bug，批量操作必须明确影响。
- Done 后的发布和部署：作为“交付”区或独立 Tab，不在审查页面末尾继续堆表单。

### 3.5 工作流必须补齐的反馈

- 页面顶部显示最近更新时间和“正在同步”状态，避免用户不知道轮询是否还在工作。
- 每个运行阶段展示开始时间、耗时、attempt、执行器和失败原因。
- 回滚、删除、驳回等高风险动作使用 Dialog，说明会影响哪些 Artifact、分支和工作副本；不要只用原生 `window.confirm`。
- 对重复点击、离开页面、网络中断和本地 Agent 未连接提供可恢复提示。
- 人工审查决策记录操作者、时间、意见和当时的 findings 快照。
- 当前阶段不可访问时仍允许查看历史产物，不应因为主状态变化而把已有内容隐藏。

## 4. 视觉与交互基线

现有 `PageHeader`、`ListToolbar`、`RecordListItem`、`EmptyState`、`WorkflowSteps`、`StageCard` 等组件足以支撑第一轮改造，优先补齐复合能力：

- `Pagination`：受控的 page/pageSize、无障碍页码、loading 保持布局。
- `ListQueryState`：统一 URL 解析、默认值、筛选变更重置页码和个人偏好。
- `DataListLayout`：标题、指标、工具栏、结果计数、列表、空态、错误态和分页。
- `StatusBadge`：统一业务状态文案、颜色和可访问 label。
- `WorkflowActionBar`：单主动作、次级动作、更多菜单、阻塞说明和提交态。
- `WorkflowStageRail`：紧凑阶段时间线，支持键盘和 URL 定位。
- `DecisionPanel`：产物预览、意见输入和确认/驳回的固定操作区。
- `AuditTimeline`：历史状态、会话、Artifact、Evidence 和人工决策。

视觉上继续保持现有克制的 B2B 管理台方向：中性画布、轻边框、有限状态色和较高信息密度。不要通过增加卡片、彩色 Badge 或大面积背景来掩盖信息架构问题。

需要先解决规范与实现的事实冲突：`docs/frontend-shadcn-design-spec.md` 声明主色为蓝色、字体为 Avenir Next，但 `apps/web/src/globals.css` 和 `design-tokens.ts` 实际使用黑色主色、Inter 字体。应选择一个 source of truth，并同步文档、token 和视觉回归基线。

## 5. 分阶段实施路线

### P0：先让系统可用

1. 定义统一 `PageResult<T>` 和列表 query DTO，改造工作流、需求、问题、缺陷四类高频列表为服务端分页。
2. 将搜索、筛选、排序、分页全部纳入 URL，补通用 `Pagination` 和 `ListQueryState`。
3. 给列表补“无数据 / 无匹配 / 加载 / 错误 / 权限不足”五类状态。
4. 工作流详情顶部固定当前状态和唯一主动作，隐藏或收拢非当前阶段动作。
5. 将工作流阶段、当前 Tab 和 Diff 选择写入 URL；刷新和返回不丢失上下文。

### P1：建立可持续的工作台

1. 上线“我的工作队列”，把待确认、执行中、失败、审查待处置聚合起来。
2. 将工作流详情拆成“当前阶段 / 变更与审查 / 上下文 / 历史”四个视图。
3. 把本地执行、OpenDesign 和云端执行统一为 Execution Session 视图，明确领取、进行中、回传、完成、失败生命周期。
4. 统一列表页骨架、状态 Badge、错误反馈、确认 Dialog 和按钮优先级。
5. 为工作流和审查补 Playwright 级别的关键路径验证，至少覆盖刷新、返回、失败重试和人工决策。

### P2：提高团队协作效率

1. 支持保存筛选视图，例如“我的待确认”“本周失败”“某项目未关闭缺陷”。
2. 支持关注/订阅工作流和异常通知，减少用户主动轮询。
3. 在项目层提供需求、工作流、质量和排期的关联视图。
4. 通过权限和角色定制队列、主操作和数据范围，而不是所有用户看到同一套入口。
5. 增加视觉回归、无障碍检查和列表性能预算，避免后续页面重新分叉。

## 6. 验收标准

### 列表页

- 首屏请求不返回无界全量业务列表。
- 复制当前 URL 给另一位用户，能得到同样的筛选和排序结果。
- 从详情返回列表后，保持原页码和筛选条件。
- 筛选无结果时，用户能一键清除筛选；空数据时能得到下一步建议。
- 改变筛选后不会继续展示上一页的旧数据。

### 工作流页

- 用户在首屏能看到当前状态、当前阶段、阻塞原因和唯一主操作。
- 任意阶段都能在不通读整页的情况下完成核心动作。
- 刷新后保留当前 Tab、阶段和 Diff 文件位置。
- 运行、失败、等待确认、人工审查和完成态都有不同且可执行的反馈。
- 所有破坏性操作说明影响范围，并记录可追溯的决策历史。

### 产品体验

- 新用户从工作队列进入待处理任务，不需要先理解全部导航。
- 研发人员能从需求一路追到工作流、分支、Artifact、审查和发布。
- 管理者能从项目看到异常，而不是逐条打开工作流寻找问题。
