# FlowX 用户使用手册

本文档面向产品、研发负责人、项目经理和业务协作同学，帮助你在 FlowX 中从需求录入一路推进到 AI 执行、人工评审、问题沉淀与发布。

## FlowX 工作方式

FlowX 把研发流程拆成可中断、可确认的阶段，核心目标是：

- 让 AI 先产出，再由人确认关键节点
- 每一阶段都可追溯，降低黑盒风险
- 统一管理需求、代码仓库、执行结果、审查结论

主流程是：

`工作区 -> 项目 -> 需求 -> 一条工作流（仓库上下文、产品构思、设计、Demo、任务拆解、技术方案、开发、AI 审查） -> 人工决策 -> 问题/缺陷沉淀（可选部署）`

产品构思、设计和 Demo 不是需求提交前的独立流程，而是同一条工作流中的可选阶段。是否执行、跳过或退回，由工作流详情页当前阶段的操作决定。

## 功能菜单

左侧导航主要有以下模块：

- `工作区`：管理业务边界与代码仓库
- `项目`：承接实际交付单元
- `需求`：录入需求、验收标准，发起流程
- `工作流`：查看和推进每条研发流程
- `问题项`：沉淀 AI 审查里的改进项（Issue）
- `缺陷`：沉淀明确 Bug（Bug）
- `AI 凭据`：配置你个人的 Cursor / Codex 凭据
- `API Token`（设置 → `/settings/api-tokens`）：生成本机 OpenDesign / MCP 用的长期 Personal API Token（`fxpat_…`）

## 本地 Agent 与 OpenDesign

若要在本机用 Cursor / Codex「本地启动」，或通过 OpenDesign 做本地构思与设计，需要安装 FlowX 本地 Agent，并配置 MCP。

**推荐**：在「设置」→ [API Token](/settings/api-tokens) 生成 Personal API Token，执行 `flowx-local login --token …` 写入 `~/.flowx/credentials.json`，再用 MCP `flowx_list_tasks` → `flowx_bind_workflow` 领取任务；构思提交后可在**同一会话**继续设计 handoff，无需再点 Web「打开本地 OpenDesign」。

Web 页面上的 `打开本地构思` / `打开本地 OpenDesign` / `回传本地设计` 仍可作为未配置长期 token 时的可选兜底。

平台内完整说明见侧栏「设置」→ **[本地 Agent](/local-agent)**（或直接打开 `/local-agent`）。

快速安装启动：

```bash
npm install -g @flowx-ai/local --registry https://registry.npmjs.org
flowx-local setup
flowx-local login --token fxpat_…
flowx-local serve
```

## 快速上手

### 步骤 1：登录系统

在登录页可选两种方式：

- 账号密码登录（也支持注册后登录）
- 钉钉登录（适合企业组织身份接入）

### 步骤 2：配置个人 AI 凭据（推荐）

进入 `AI 凭据` 页面，按需配置：

- `Cursor API Key`
- `Codex / OpenAI API Key`

保存后，系统在工作流执行时会优先使用你的用户凭据。

### 步骤 3：创建工作区并收录仓库

进入 `工作区` 页面：

1. 新建工作区（例如“支付平台”）
2. 在工作区下新增代码仓库（填写仓库名称与地址）
3. （可选）配置仓库默认分支与部署模板

> 建议先把真实会改动的仓库纳入工作区，后续需求和工作流才能准确拿到上下文。

### 步骤 4：创建项目

进入 `项目` 页面，新建项目并选择归属工作区。

一个工作区下可以有多个项目；一个项目下可以有多条需求。

### 步骤 5：创建需求

进入 `需求` 页面，点击“新增需求”，填写：

- 所属项目
- 需求标题
- 需求描述
- 验收标准
- 影响仓库范围（可留空，留空表示继承工作区默认仓库）

## 从需求到交付

### 创建需求并启动工作流

需求页只负责准备研发输入并启动工作流，不需要先完成额外的前置确认或进入独立流程：

1. 在需求页点击 `新增需求`，填写所属项目、标题、描述、验收标准和可选的目标仓库范围。
2. 创建后，在需求列表或需求详情点击 `启动工作流`。
3. 在启动窗口选择 AI 执行器和本次工作流的仓库范围；不选择时使用需求默认范围。
4. 启动后进入工作流详情页。仓库上下文准备完成后，按阶段推进产品构思、设计方案、Demo、任务拆解、技术方案、开发执行和 AI 审查。

一次研发过程对应一条工作流；如需重新发起，系统会保留历史记录，并检查同一仓库范围是否已有进行中的工作流。日常操作以当前工作流详情页为准。

### 使用本地 OpenDesign 构思与设计

如果当前工作流需要在本地 OpenDesign 完成产品构思与 UI/交互设计：

1. 安装本地 Agent，并执行 `flowx-local setup`（详见 [本地 Agent](/local-agent)）。
2. 在「设置」→ [API Token](/settings/api-tokens) 生成 Personal API Token（`fxpat_…`），再执行 `flowx-local login --token …`（或设置 `FLOWX_API_TOKEN`）。
3. 在 Cursor / Codex 启用 `flowx-local mcp`：`flowx_list_tasks` → 确认工作流 → `flowx_bind_workflow`。
4. 构思：`flowx_get_brainstorm_handoff` → 澄清并写 `spec.md` → `flowx_submit_brainstorm`（响应会提示进入 design）。
5. **同一会话**立刻 `flowx_get_design_handoff` → 完成设计 → `flowx_submit_design`（也可在工作流详情点击 `回传本地设计`）。
6. 平台进入 `待确认设计方案`，由人工确认、驳回或跳过后继续下一阶段。

**可选兜底**：未配置长期 token 时，可在工作流详情点击 `打开本地构思` / `打开本地 OpenDesign` 写入短期会话；金路径下不需要第二次点击「打开本地 OpenDesign」。

网络中断时完成报告会进入本地 Outbox，可通过 `flowx-local sync` 重试。详细配置和结果格式见
[OpenDesign 本地设计阶段](opendesign-design-stage.md)。

### 在 Cursor Agent 中启用 FlowX MCP

MCP 要配置在实际运行 Agent 的工具里，不是配置在 OpenDesign 应用里。如果你使用 Cursor Agent，请在 Cursor 的 MCP 配置中登记 `flowx-local`：

```json
{
  "mcpServers": {
    "flowx": {
      "command": "flowx-local",
      "args": ["mcp"]
    }
  }
}
```

先安装并配置本机 Agent 与 Personal API Token：

```bash
npm install -g @flowx-ai/local --registry https://registry.npmjs.org
flowx-local setup
flowx-local login --token fxpat_…
flowx-local serve
```

Cursor 会按需启动 `flowx-local mcp`。配置 PAT 后，它通过 `~/.flowx/credentials.json` 与 `current-workflow.json` 鉴权并绑定任务；也可兼容读取 Web 写入的短期 `active-design` 会话。通过 FlowX Web 的「本地启动」打开 Cursor 时，`flowx-local` 还可能写入项目级 `.cursor/mcp.json`。手工配置时不要把 `FLOWX_API_BASE_URL` 写死为 `127.0.0.1`，也不要把 `~/.flowx` 下的凭据、会话文件或 token 提交到 Git。

### 启动研发工作流

可以在需求列表或需求详情里启动工作流。启动时可选本次执行的仓库范围：

- 不选：使用需求默认仓库范围
- 手动选择：只推进本次要并行处理的仓库

### 在工作流详情推进每个阶段

工作流阶段按实际配置和是否跳过可选阶段推进，通常是：

1. 仓库 Grounding：同步仓库并生成代码上下文
2. 产品构思：本地构思或 AI 生成产品规格，可跳过；若已进入设计阶段，可在「产品构思」面板点击「重新构思」回到构思（设计产物保留对照），再用 MCP 或（可选）打开本地构思重写规格
3. 设计方案：本地 OpenDesign 回传或生成设计稿，可确认、驳回或跳过；推荐构思提交后同一会话继续设计，Web「打开本地 OpenDesign」为可选兜底
4. Demo 页面：生成可预览 Demo，可确认或跳过
5. 任务拆解（Task Split）
6. 技术方案（Technical Plan）
7. 开发执行（Execution）
8. AI 审查（AI Review）
9. 人工评审（Human Review）

在各阶段你可以进行的常见操作：

- 执行阶段（如“执行任务拆解”“执行开发”“执行 AI 审查”）
- 确认 / 驳回（用于需要人工把关的节点）
- 人工修改（直接修正阶段产出）
- 填写意见（用于退回或指导下一轮）
- 重新构思（设计阶段时，在产品构思面板将工作流退回构思并保留设计产物）

AI 审查完成后，人工决策通常有三种：

- `通过`：工作流进入完成态
- `退回开发执行`：回到执行阶段继续修复
- `回滚`：终止当前结果并回退

### （可选）提交并推送、触发部署

当人工确认通过后，可在工作流内执行：

1. `提交并推送到远程`
2. `按仓库触发部署`

部署会读取仓库级部署模板。如果按钮不可用，通常是仓库未开启部署配置或当前工作流无可部署仓库。

## 状态与进度

### 工作流状态（常见）

- `待仓库 grounding`
- `待产品构思`
- `待设计方案` / `待确认设计方案`
- `待 Demo 页面` / `待确认 Demo`
- `待任务拆解` / `待确认任务拆解`
- `待生成方案` / `待确认技术方案`
- `待执行开发` / `开发执行中`
- `待 AI 审查`
- `待人工评审`
- `已完成` / `失败`

### 阶段状态（常见）

- `待执行`
- `执行中`
- `已完成`
- `待人工确认`
- `执行失败`
- `已驳回`

## 问题、缺陷与追踪

- 在 `问题项` 页面追踪改进项（Issue）
- 在 `缺陷` 页面追踪明确故障（Bug）
- 每条记录都可追溯到来源需求、工作流和审查上下文，便于复盘和闭环

## 常见问题

### 为什么我不能直接“启动研发工作流”？

请先检查需求是否属于当前组织、工作区是否配置了可用仓库，以及同一需求和仓库范围是否已有进行中的工作流。工作流启动后，产品构思、设计和 Demo 等可选阶段会在工作流详情中处理，不需要提前完成额外的确认流程。

### 为什么“触发部署”失败或不可用？

优先检查：

- 当前工作流是否有可部署仓库
- 对应仓库是否在工作区里配置了部署模板并启用
- 是否已先执行“提交并推送到远程”

### 我配置了 AI 凭据，为什么没生效？

请确认：

- 你当前登录账号与配置凭据的是同一账号
- 页面状态显示“已配置”
- 工作流是凭据配置后新启动的（旧流程可能仍使用旧上下文）

## 团队落地建议

建议团队按下面顺序落地：

1. 管理员先完成工作区、仓库、部署模板初始化
2. 各成员配置自己的 `AI 凭据`
3. 项目负责人按项目组织需求
4. 创建需求后直接发起一条工作流，在工作流详情中按阶段确认和推进
5. 在 AI 审查后坚持人工决策与问题沉淀

这样可以把“AI 产出”和“人类决策”稳定地串成可复用流程。
