# Workflow HTML Artifacts

FlowX stores human-readable stage deliverables as versioned HTML files under `.flowx-data`. SQLite keeps pointers and metadata only; the HTML file is the canonical view for operators.

Slice 1 covers the **TECHNICAL_PLAN** stage (`plan.html`). Other stages may follow the same pattern later.

## Directory layout

Artifacts live beside existing workflow clones:

```text
.flowx-data/workflows/{workflowRunId}/
├── repositories/              # existing Git clones
└── artifacts/
    ├── manifest.json          # stage → current version/paths
    ├── shared/                # optional shared assets (e.g. flowx-artifact.css)
    └── plan/
        └── v{n}/              # n = StageExecution.attempt for TECHNICAL_PLAN
            ├── plan.html      # server-rendered HTML document
            └── plan.meta.json # machine-readable plan sidecar
```

Root resolution: `getWorkflowArtifactsRoot()` in `apps/api/src/workflow/workflow-artifact.paths.ts`. Override the base with `FLOWX_ARTIFACTS_ROOT` (tests use this for temp dirs).

Each plan generation writes a new `v{n}` directory. `manifest.json` always points at the latest plan version.

## File formats

### `manifest.json`

Maps each artifact kind to its current on-disk location:

```json
{
  "plan": {
    "version": 1,
    "path": "plan/v1/plan.html",
    "metaPath": "plan/v1/plan.meta.json",
    "sha256": "<hex digest of plan.html bytes>",
    "confirmedAt": null
  }
}
```

After human confirmation, `confirmedAt` is set to an ISO-8601 timestamp and `plan.meta.json` status becomes `CONFIRMED`.

### `plan.meta.json`

Structured plan data for execution and tooling (not the HTML parse tree):

```json
{
  "summary": "string",
  "implementationPlan": ["string"],
  "filesToModify": ["string"],
  "newFiles": ["string"],
  "riskPoints": ["string"],
  "status": "WAITING_HUMAN_CONFIRMATION",
  "confirmedAt": null
}
```

`status` is one of `WAITING_HUMAN_CONFIRMATION`, `CONFIRMED`, or `REJECTED`.

### `plan.html`

Self-contained HTML generated by `renderPlanHtml()` from validated `GeneratePlanOutput` JSON. Sections: 摘要, 实施步骤, 涉及文件, 新增文件, 风险点, plus a footer with run id, version, and status. All dynamic text is HTML-escaped; the template includes no `<script>` tags.

## API

### `GET /workflow-runs/:id/artifacts/plan`

| Item | Detail |
|------|--------|
| Auth | Same session guard as other workflow routes (`Authorization: Bearer <token>`) |
| Success | `200`, `Content-Type: text/html; charset=utf-8`, body is `plan.html` |
| Errors | `404` when no plan artifact exists or the file is missing on disk |

Implementation: `WorkflowController.getPlanArtifact` → `WorkflowService.readPlanArtifactHtml` → `WorkflowArtifactService.readPlanHtml` (reads `manifest.json`, then loads the HTML path).

`.flowx-data` is **not** exposed as a public static mount; artifacts are only served through this authenticated endpoint.

## Stage output pointer

When plan generation succeeds, `WorkflowService` writes artifacts and merges a backward-compatible `_artifact` pointer into `StageExecution.output` for `TECHNICAL_PLAN`:

```json
{
  "summary": "...",
  "implementationPlan": [],
  "filesToModify": [],
  "newFiles": [],
  "riskPoints": [],
  "_artifact": {
    "kind": "plan",
    "version": 1,
    "htmlPath": "plan/v1/plan.html",
    "metaPath": "plan/v1/plan.meta.json",
    "sha256": "..."
  }
}
```

Top-level plan fields are retained so existing JSON views (e.g. `StageCard`) keep working during migration.

If artifact write fails, the stage still completes with DB-only plan data; the error is logged and `_artifact` is omitted.

On confirm, `confirmPlanArtifact` updates `plan.meta.json` and `manifest.json`. Execution (`runExecution`) prefers confirmed meta via `loadPlanMeta`, falling back to the `Plan` table when meta is absent.

## Web preview

On the workflow detail page, when **TECHNICAL_PLAN** is selected and stage output contains `_artifact`:

1. `api.fetchPlanArtifact(workflowRunId)` fetches HTML with the Bearer token (iframe cannot set headers).
2. The response string is stored in component state.
3. An iframe renders it via `srcDoc={planHtml}` under a **方案预览** section.

Relevant code: `apps/web/src/api.ts` (`fetchPlanArtifact`), `apps/web/src/pages/WorkflowRunDetailPage.tsx`.

This avoids putting tokens in iframe URLs.

## Security

| Control | Detail |
|---------|--------|
| Server-rendered HTML | AI output is validated as JSON at the boundary; HTML is produced only by `renderPlanHtml()` with `escapeHtml()` on all dynamic fields |
| No scripts in template | Plan HTML is static structure + escaped text; no `<script>` in the generator |
| Sandboxed iframe | `sandbox=""` (no `allow-scripts`, no `allow-same-origin`) — stricter than the Demo preview iframe |
| Authenticated fetch | Artifact endpoint requires session auth; HTML is loaded via `fetchPlanArtifact`, not a public URL |

## Deployment assumption: single-node disk

Artifacts are read and written on the API host's local filesystem. There is no shared object store or replication in slice 1.

**Implication:** In a multi-instance API deployment, only the node that wrote the artifact can serve it unless `.flowx-data` is on shared storage mounted at the same path on every instance. For production scale-out, plan for shared volume or object storage in a follow-up.

Artifact write failure is non-fatal: the workflow stage succeeds with SQLite/Prisma plan data as fallback.
