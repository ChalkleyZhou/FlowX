# Commit 驱动的项目变化简报设计

## 背景

当前项目简报以 commitlint 类型组织内容，正文主要是 `feat`、`fix`、`docs`、`refactor` 等提交集合。它适合研发人员回顾代码活动，但产品、测试或项目相关人员仍需自行理解技术描述，难以快速知道当天发生了哪些项目变化。

本设计将简报从“研发提交日报”调整为“项目变化简报”。首版仍然只使用 commit 作为事实来源，不要求需求、任务或缺陷已经录入 FlowX，也不依赖 MR、Issue、Pipeline、Release 或部署数据。

## 目标

- 将相关 commits 聚合成少量、可阅读的项目变化主题。
- 使用非研发角色也能理解的语言描述 commit 已明确表达的变化。
- 保留原始 commits 作为证据，所有摘要均可追溯。
- 对证据不足的信息明确标记为待确认，不通过 AI 补全事实。
- 保持当前简报生成、存储和投递流程不变，仅调整内容生成与展示语义。

## 非目标

- 不判断某项变化是否可测试、已上线、已发布或已验收。
- 不生成验证场景、测试建议、潜在风险或排期影响。
- 不区分“产品关注点”和“测试关注点”。
- 不依赖 FlowX 内的需求、工作流、Bug、排期或部署数据。
- 不在首版扩展新的数据源。

## 设计原则

### Commit 是唯一事实源

项目变化主题、摘要、涉及模块和待确认事项都必须能由输入 commits 支持。AI 可以整理、合并和改写，但不能补充 commit 中没有出现的业务背景、用户对象、状态或结论。

### 主题优先，提交分类退居附录

正文按“同一件项目变化”聚合，而不是按仓库或 commit 类型分组。`feat`、`fix` 等分类只用于辅助理解和底部研发记录，不决定正文结构。

### 不确定性显式呈现

当 commit 描述只能确认技术动作，不能确认业务含义时，正文应保留技术事实，并把缺失的信息写入待确认事项。禁止为了让简报看起来完整而强行推断。

## 信息架构

简报采用两层结构：顶部用于快速阅读，底部用于追溯。

```text
项目名 · 项目变化简报 · 日期

今日概览
1-2 句话概括当天可确认的主要变化。

主要变化
1. 主题名称
   变化摘要
   涉及模块（仅可确认时展示）
   依据：关联 commits

2. 主题名称
   ...

待确认事项
- commit 描述中缺失或含义不明确的信息

研发记录
- 按 feat、fix、docs、refactor 等类型展示原始 commits
```

### 今日概览

- 只总结已形成主题的内容。
- 不使用“完成上线”“可测试”“解决了用户问题”等无法由 commit 证明的表述。
- 无有效 commit 时显示“今日暂无可归纳的项目变化”。
- 有 commits 但无法形成可靠主题时，说明“今日有研发提交，但现有描述不足以归纳项目变化”。

### 主要变化

每个主题包含：

- `title`：简短、面向项目的主题名称。
- `summary`：说明发生了什么，不展开实现细节，不虚构业务价值。
- `modules`：从仓库名、commit scope 或明确描述中提取；无法确认时为空。
- `commitReferences`：关联 commit 的稳定标识和原始标题。

主题不包含验证建议、风险判断或交付状态。

### 待确认事项

仅在输入确实存在信息缺口时输出，例如：

- `优化登录处理` 没有说明具体改变了什么行为。
- `调整接口` 没有说明影响哪个调用方或业务流程。
- 多个 commit 可能相关，但描述不足以确认是否属于同一变化。

待确认事项应指出“缺少什么信息”，不提出无依据的猜测。

### 研发记录

保留当前 commit 分类结果和原始提交标题，作为完整证据附录。正文主题生成失败时，研发记录仍应正常输出。

## 主题聚合规则

聚合时按以下证据从强到弱判断相关性：

1. commit scope 或明确模块名称一致。
2. commit 标题包含相同功能、页面、接口或业务对象。
3. 一个提交描述新增能力，另一个明确描述对该能力的修复或补充。
4. 仅 commit 类型相同不能作为聚合依据。

一个 commit 可以只归属于一个正文主题，避免重复计算。无法可靠归组的 commit 可以形成独立主题，也可以只出现在研发记录中；选择取决于其描述是否足以形成有意义的变化摘要。

以下提交默认不形成独立业务主题，除非描述明确体现项目变化：

- merge commit
- 纯格式化或 lint 调整
- 依赖锁文件刷新
- 无明确影响的 chore
- 信息量过低的标题，如 `update`、`fix`、`调整代码`

这些提交仍保留在研发记录中。

## AI 输出契约

建议将当前按 `features/fixes/risks/otherNotes` 划分的结构替换为面向主题的结构：

```ts
interface ProjectChangeBriefingSummary {
  headline: string;
  summaryParagraph: string;
  topics: Array<{
    title: string;
    summary: string;
    modules: string[];
    commitReferences: Array<{
      repository: string;
      title: string;
    }>;
  }>;
  openQuestions: string[];
}
```

输出约束：

- `topics` 中的每个结论必须关联至少一个输入 commit。
- `commitReferences` 必须来自输入，不允许生成或改写成不存在的提交。
- `modules` 只允许使用输入中明确出现的仓库、scope 或模块名称。
- `openQuestions` 可以为空，不要求为了完整性生成问题。
- 禁止输出上线、发布、测试、验收、风险、排期和用户反馈结论。
- 禁止用“可能提升”“预计改善”等措辞虚构价值。

## 数据流

1. 收集日期窗口内的 push commits。
2. 规范化 commit 标题、仓库、分支、作者和可用标识。
3. 过滤不适合进入正文主题的低信息提交，但保留完整研发记录。
4. 将可用 commit facts 发送给 AI，生成主题化结构。
5. 校验每个主题引用的 commit 和模块均存在于输入 facts。
6. 渲染“今日概览、主要变化、待确认事项、研发记录”。
7. 沿用现有 Markdown、HTML、存储和投递流程。

## 降级与异常处理

- AI 未启用、调用失败或输出 schema 校验失败：正文退化为确定性的 commit 分类摘要。
- AI 引用了不存在的 commit：视为整次结构化摘要无效并降级，不局部接受。
- 没有 commit：输出空状态，不调用 AI。
- 只有低信息 commit：不强行生成主题，输出说明和研发记录。
- 多仓库存在同名模块：模块展示同时保留仓库上下文，避免错误合并。

降级结果不应包含待确认事项的自动推断，因为确定性逻辑无法可靠判断缺失的业务语义。

## 展示与命名

- 标题由“研发日报”调整为“项目变化简报”。
- 页面、Markdown 和 HTML 使用相同章节顺序和字段语义。
- Git 证据在正文中保持简洁，在研发记录中展示完整标题和仓库。
- 不新增按角色切换的产品版或测试版。

## 测试策略

### 主题生成契约

- 多个描述同一功能的 commits 被聚合为一个主题。
- 相同 commit 类型但内容无关的 commits 不被错误合并。
- 低信息 commit 不形成虚构主题。
- 输出不能引用输入中不存在的 commit 或模块。
- 输出不能包含可测试、上线、验收、风险或排期结论。

### 渲染

- 有主题时按设计顺序渲染全部章节。
- `modules` 或 `openQuestions` 为空时不渲染空标题或占位项。
- 无 commit、只有低信息 commit、AI 降级三种状态均有明确文案。
- Markdown 与 HTML 表达一致，特殊字符继续正确转义。

### 回归

- 原始 commits 仍完整出现在研发记录中。
- 简报生成、重新生成、历史查询和投递行为不变。
- AI 不可用时简报仍能生成和发送。

## 验收标准

- 非研发读者可以先看到当天少量项目变化主题，而不是一组 commit 类型列表。
- 每个主题都能追溯到真实 commit。
- 简报不宣称 commit 无法证明的测试、发布、风险或业务结论。
- 信息不足时宁可展示待确认或只保留研发记录，也不生成看似完整的推断。
- 不录入 FlowX 需求也能生成简报。

## 后续可能方向

若未来引入 MR 描述、Issue、需求、部署或测试环境数据，可以在独立设计中扩展验证状态、风险和业务背景。本设计不为这些能力预留首版输出字段，避免当前 commit-only 模型承担无法可靠完成的职责。
